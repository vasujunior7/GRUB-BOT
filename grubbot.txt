What is Grubbot?
Grubbot is a four-stage autonomous pipeline that takes your custom tool definitions and
produces a small, reliable, local model that can use them correctly.
Most open-source finetuning tools stop after one training run. Grubbot keeps going — it tests
the model, finds exactly where it fails, generates targeted training data for those failure patterns,
retrains, and repeats until your accuracy target is hit.
The name comes from the grub's instinct: dig through everything until you find what you need.
That's the loop.
The Problem
Every developer who wants a local AI assistant hits the same wall: small models are unreliable
with custom tools. They hallucinate parameters, call the wrong tool, or format responses
incorrectly.
The existing solutions all have the same flaw — they finetune once, evaluate once, and stop. If
the model fails on 20% of your tool calls, you have no clear path to fix it other than manually
curating more data and starting over.
No clean solution exists for this specific problem. Grubbot is that solution.
The Four Stages
01
Data Generation
Calls a free LLM (Gemini Flash / Groq / local 70B) to
synthesize training examples from your tool definitions
free API or local
02
Finetuning
Trains a small local model on the generated data using
Unsloth + TRL with LoRA adapters
runs on your GPU
03
Evaluation
Runs the finetuned model against a held-out test set and
scores accuracy per tool
local
04
Failure Clustering
Embeds failed outputs with sentence-transformers,
clusters with HDBSCAN, generates targeted retraining
data, loops back to stage 02
local · autonomous
The loop between stages 02–04 is the core innovation. It runs automatically until your target
accuracy is reached or a maximum iteration count is hit.
What Makes It Different
Starts from your tools, not benchmarks
Not generic function calling datasets. Not someone else's API. Your tools, defined however you
want — YAML, Python functions, plain text.
Full autonomous loop, not one-shot
The pipeline tests, finds failures, fixes them, and retests. This is the pattern from Karpathy's
autoresearch applied to tool-use finetuning.
Failure clustering, not just accuracy scores
Grubbot doesn't just say '78% accuracy'. It groups failures by pattern using HDBSCAN on
sentence embeddings and tells you exactly what's going wrong and why.
Fully free after setup
Uses a free cloud API once for initial data generation (Gemini 2.0 Flash, Groq free tier, or a
local 70B model via Ollama). Everything after — finetuning, eval, the loop — runs entirely on
your hardware. Private. No recurring cost.

User Experience
Step 1 — Define your tools (5 min)
tools:
 - name: get_inventory
 description: Check stock levels
 parameters:
 product_id: string
 warehouse: string
Step 2 — Set your goal (2 min)
# goal.md
Target: 90%+ accuracy on tool selection and parameter filling.
Priority: never hallucinate parameters.
Step 3 — Run
grubbot run --tools tools.yaml --goal goal.md --model qwen2.5-3b
Step 4 — Wake up to results
Run complete. 3 iterations.
get_inventory → 96% ✓
send_email → 91% ✓
Failure clusters resolved:
 parameter hallucination → fixed in iteration 2
 tool confusion → fixed in iteration 3
Model saved: ./models/grubbot-qwen2.5-3b-v3

Build Phases
Phase 1 — Core Pipeline (weeks 1–6)
Working end-to-end: user defines tools → data generated → model finetuned → evaluation
report. This alone is shippable and useful.
Phase 2 — The Autonomous Loop (weeks 7–10)
Add failure clustering and the retrain loop. Model improves autonomously until target accuracy
is hit. This is where Grubbot becomes genuinely novel.
Phase 3 — The Research Layer (months 3–4)
Real usage generates real data: which failure patterns are most common, how many examples
per tool does it actually need, which model sizes work best. Write this up as a paper.
